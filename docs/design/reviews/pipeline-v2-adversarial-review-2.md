1. **BLOCKING — Round-1 finding 1 survives in §§3–3c.** The saga is not enumerable yet. Most steps do not actually specify all eight required parts; rollback contains multiple independently crashable deletions; burn omits its commit/push conflict protocol; one `upload-asset` step does not enumerate the signed artifact set; and the “whole pipeline” stops at dispatch rather than modeling run discovery, completion, failure, publication, and terminal verification.

2. **BLOCKING — Round-1 finding 2 is only partially discharged by §2.** Four values are adequate for one point observation, but not for a history. `epoch` has no implementable definition, and sequential GitHub reads cannot create a global snapshot. Example: observe release ABSENT; A2 creates it; cutter issues create. Every read was individually authoritative, yet the decision was not linearizable. ETags cannot detect every mutate-restore ABA. The protocol needs server-side CAS/idempotency or explicit read-set validation; otherwise `ABSENT` remains a transient fact, not permission to mutate.

3. **BLOCKING — Round-1 finding 3 survives in §§3 and 3b.** “Saga” is currently a template, not a temporal specification. A finite state machine can represent unbounded execution traces; switching vocabulary from states to steps does not remove the need to enumerate crash points, response-loss states, compensation substates, retry bounds, and legal transitions. `PENDING is legitimately ambiguous` is precisely the ambiguity round 1 rejected.

4. **BLOCKING — Round-1 finding 4 survives in §3 and Q4.** Release, asset, and run IDs are generated after intent signing. If the journal is irrelevant, there is no authenticated durable location holding the expected IDs. An A2 writer can also copy the public attempt ID into a release body or dispatch; equality proves correlation, not authorship. GitHub’s current dispatch API can return a run ID on a successful `200`, but a timeout after server acceptance still leaves no idempotency key or trustworthy recovery locator. [GitHub workflow-dispatch API](https://docs.github.com/en/rest/actions/workflows)

5. **BLOCKING — Round-1 finding 5 is only half discharged by §5.** The persistent local `flock` design is sound. The tag push is not yet a global ownership fence: if two cutters push the identical tag object, the second push is “up to date,” not a losing CAS, so both may advance. Git documents this as `=`—no update needed. [Git push semantics](https://git-scm.com/docs/git-push) Distinct per-execution claims would avoid identical objects, but §1 omits the attempt ID from its minimum signed intent. An A2 writer can also front-run an already-signed tag.

6. **BLOCKING — Q2 is a prerequisite, not an open implementation question.** On GitHub Free, branch/tag rulesets are available only for public repositories; private repositories require Pro, Team, or Enterprise. Rulesets can restrict tag updates and deletion when available. [GitHub ruleset availability](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-rulesets/about-rulesets), [tag rules](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-rulesets/available-rules-for-rulesets) Therefore a private Free repository cannot satisfy §5’s stated immutability premise using that mechanism. Even where rulesets exist, v2 must verify active enforcement and the bypass set before cutting.

7. **BLOCKING — Round-1 finding 6 survives through a compensation-ordering contradiction.** Recovery derives authority only from the signed tag, but rollback may delete that tag before burn is durable. A crash in that interval destroys the recovery root while leaving the version unburned. The signed intent must remain reachable until the terminal record is durably anchored—and likely forever—or burn must commit before destructive cleanup. Rollback also needs coordination with in-flight CI; “still draft” followed by deletion is vulnerable to CI publishing between the read and mutation.

8. **BLOCKING — Round-1 finding 7 survives in §§1, 3, and 3b.** Section 3b requires server-created object identities to match signed intent even though those IDs cannot be known when intent is signed. Section 1 also leaves release metadata, workflow identity/version, repository host and stable repository ID, and the exact signing/notarization claims under-specified. `owner/name` is weaker than a stable GitHub repository identity if rename/recreation is in scope. A current object’s ID cannot reveal whether an earlier receipt was deleted and replaced unless an independent authenticated receipt records the original ID.

9. **BLOCKING — Round-1 finding 8 survives in §7.** The signer-policy alternative is self-contradictory: policy stored only in the unprotected release-commit tree is not “out of band.” A2 can commit a policy naming its own key unless that policy digest or ancestry is anchored independently. A1 is also defined too narrowly: the plan must state whether checkout files, executables, environment, GPG keyrings, credentials, and same-UID processes are trusted. A2 needs an exact permission matrix, including workflow edits, release edits, ref deletion, ruleset administration, and token identity. The asynchronous CI publisher is an additional actor whose composition with rollback is unmodeled.

10. **BLOCKING — Round-1 finding 9 survives in new clothing.** Section 1’s minimum intent omits `attempt-id`, while §§3 and 6 assume it is signed into the intent. More fundamentally, a public attempt ID is not an authorization credential. The design must distinguish immutable release intent from a unique execution claim and specify what gives that claim ownership. Q3 must require confirmation of canonical frozen bytes, including the commit and artifact digests, with revalidation immediately before signing; a vague preview does not close A7.

11. **BLOCKING — Round-1 finding 10 survives at the remote-receipt layer.** V2 correctly removes authority from the local journal, but then treats A2-writable releases, assets, bodies, and workflow inputs as authoritative receipts. That changes the location of the mutable authority without supplying authenticity or monotonicity. The journal-irrelevance property can pass while every remote receipt is attacker-created. Post-effect identities need an authenticated append-only receipt mechanism, verifiable actor attestation, or desired-state semantics that do not claim attempt ownership.

12. **BLOCKING — Q6’s chained-log proposal is insufficient.** A signed hash chain on an unprotected `main` detects deletion only for an observer holding a trusted later checkpoint. An attacker can reset `main` to an older valid signed prefix; a fresh cutter sees a perfectly valid chain and concludes the missing burn never existed. Therefore version revocation against A2 is unachievable using only that mutable repository state. It needs a protected immutable ref, external transparency/append-only service, trusted monotonic checkpoint, or another non-rollbackable anchor. GitHub’s distinct immutable-release control protects tags/assets only after publication and only when enabled; ordinary publication is not intrinsically immutable. [GitHub immutable releases](https://docs.github.com/en/enterprise-cloud@latest/code-security/concepts/supply-chain-security/immutable-releases)

13. **BLOCKING — Q1–Q6 defer architecture decisions that must precede code.** Q2–Q6 are directly load-bearing. Q1’s carrier choice can remain flexible, but canonical serialization, domain separation, digest algorithm, trust-root anchoring, and blob reachability cannot. Q7 is legitimately post-design. Missing questions include: where post-effect receipts are authenticated; how a unique execution claim is acquired; how rollback fences or cancels CI publication; what exact A2 permissions exist; how signer-policy rotation works; and what complete dispatch→run→publication state machine is authoritative.

14. **NON-BLOCKING — several v2 changes are directionally correct and should be retained.** Journal irrelevance, persistent-FD locking, four-way fail-closed observations, schema reuse, explicit recovery, and rule-level mutation tests address real round-1 defects. Verification should additionally inject a crash after every remote mutation, same-object tag pushes, lost successful responses, ABA mutations, burn-log truncation, and rollback-versus-publication schedules.

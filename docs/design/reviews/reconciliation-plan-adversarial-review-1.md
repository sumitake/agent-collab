1. **BLOCKING — R1 is not yet enumerable as written.** The arithmetic already disagrees: `9 + 3 + 1 = 13` journal states, plus absent means 14 rows, while the plan also says “twelve states plus absent.” More importantly, “evidence” is not one predicate per source: tag ref, annotated-tag OID, peeled commit OID, signature validity, signer authorization, release ID/state, and each asset’s ID/digest are distinct facts and cross-source relations. An implementer would improvise which identity and object type each cell means.

2. **BLOCKING — the missing bypass class is indeterminate-as-absent.** The unasked column is: “Was this fact authoritatively observed, and from what coherent observation epoch?” GitHub `404`, permission masking, timeout, rate limit, incomplete pagination, stale cache, malformed response, and genuine absence must not collapse into “absent.” Every predicate needs at least `PRESENT`, `ABSENT`, `UNKNOWN`, and `CONFLICT`; only authoritative `ABSENT` may satisfy a forbidden-evidence rule.

3. **BLOCKING — a flat matrix handles quiescent invariants, not the complete protocol.** Concurrent cutters and GitHub mutations create unbounded interleavings, and sequential API calls can produce a combination that never existed simultaneously. Keep the matrix as a finite invariant table, but add a temporal saga specification: precondition, operation/attempt ID, effect, durable receipt, postcondition, compensation, retry rule, and revalidation point. Unknown or cross-epoch evidence must stop rather than select a matrix row.

4. **BLOCKING — R1/R8 leave every pending state dangerously ambiguous.** “May or may not have landed” is not an executable rule. For each effect, the contract must say how to distinguish confirmed-not-started, confirmed-completed-by-this-attempt, completed-by-another-attempt, ambiguous, and conflicting. Without an attempt-bound idempotency key or receipt, an implementer will either replay uploads/publication/deletion or accept somebody else’s effect.

5. **BLOCKING — R5 reproduces split ownership unless unlinking is prohibited.** `flock` protects an inode, not a pathname. If an operator “clears” the file while holder A retains the old inode, holder B can create and lock a new inode; both then hold valid locks. The lockfile should remain persistent, and “clear” must mean proving/acquiring the lock—not unlinking it. A local `flock` also does nothing against a cutter in another clone or host, so global serialization still needs a remote fencing/CAS mechanism.

6. **BLOCKING — R2 delegates the most dangerous decision to an unspecified `--recover`.** The plan does not define what recovery authenticates, what operator intent authorizes adoption, whether it may mutate, or how repository/version/channel/commit are bound. An implementer could reproduce silent adoption behind an explicit flag. Recovery needs its own decision protocol and must reject ambiguous or context-mismatched signed state.

7. **BLOCKING — R3 and R4 specify presence and identity but not semantic agreement.** `ROLLED_BACK` mentions release and asset absence but does not settle the tag, duplicate/multiple matches, or recreation after rollback. Recorded identity also needs exact type and meaning: tag-object OID versus peeled commit, release ID versus tag name, asset ID plus digest/size, and expected draft/published state. A stable identity matching the journal is insufficient unless all attributes match a signed release intent.

8. **BLOCKING — A1/A2/A3 are not a closed attacker model.** Capabilities can compose: local tampering plus repo writing is stronger than either row alone. Also missing are a concurrent honest cutter, replay of a valid signature from the wrong repository/release context, local trust-anchor or Git configuration substitution, and a signing-oracle attack where the adversary cannot extract the key but steers the legitimate cutter into signing attacker-chosen intent. “Cannot forge a signature” is therefore weaker than the plan assumes.

9. **BLOCKING — consistency is being conflated with authorization and causality.** Three mutually matching sources can consistently describe the wrong release, wrong channel, replayed signed tag, or artifacts created by another attempt. The contract needs an immutable release-intent/attempt identifier binding repository, version, commit, channel, expected artifacts, and signer policy. Otherwise reconciliation proves only coincidence, not that this cut owns or is authorized to resume the effects.

10. **BLOCKING — the matrix is partly treating the journal architecture’s symptom.** A fully attacker-writable journal cannot safely be the authority selecting weaker guards; at most it is a progress hint. The root design should be a resumable, idempotent release saga whose authorization comes from signed intent and whose progress is established by attempt-bound remote receipts. The matrix remains valuable as a derived invariant specification, but it cannot repair an authoritative mutable journal by itself.

11. **NON-BLOCKING — the verification obligation overclaims precision.** “Deleting one row must fail exactly one subtest” is brittle: a sound invariant may intentionally affect several scenarios. Require schema-level completeness, per-rule mutation detection, property tests over observation outcomes, and scheduled concurrency/fault traces; do not require exactly one failing test.

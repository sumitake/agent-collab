### Fixed
- Notarization is now verified with the code-object-native `codesign --verify --strict --test-requirement '=notarized'` instead of `spctl --assess --type execute`, at BOTH the release CI gate (`verify_runtime_release.py`) and runtime activation in the client (`runtime_client.py`). The runtime entrypoint is a bare command-line Mach-O, not a `.app` bundle, and `spctl --type execute` only assesses `.app` bundles — it always rejects a bare executable ("the code is valid but does not seem to be an app"), a false negative that would fail every otherwise-valid activation release at the notarization step AND every client activation. The `notarized` codesign requirement binds to the CDHash and still rejects unsigned, ad-hoc, and Developer-ID-signed-but-not-notarized binaries by exit code; `--check-notarization` is deliberately not combined (it would fail open). Both call-sites fail closed on tool error. Note: a bare Mach-O cannot have a notarization ticket stapled, so `=notarized` relies on the host's local notarization trust state — an activation-time online/cache dependency tracked as a pipeline follow-up.
